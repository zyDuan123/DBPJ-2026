# 校园活动报名系统前端开发文档

## 1. 前端目标

前端一期目标是支撑三类角色完成完整演示闭环：学生发现并报名活动，组织者发布并管理活动，管理员审核活动并维护基础资源。页面设计应以清晰、高效、低学习成本为主，不做营销式首页。

## 2. 技术选型

- 核心框架：Vue 3 + Composition API
- 构建工具：Vite
- UI 组件库：Element Plus
- 路由：Vue Router
- 状态管理：Pinia
- 网络请求：Axios
- 日期处理：Day.js
- 二维码：Vue QRCode 或同类轻量二维码组件
- 图表：ECharts，用于管理员统计页

## 4. 路由与权限

登录成功后根据 `role` 跳转到不同入口：

- `STUDENT`：`/student/activities`
- `ORGANIZER`：`/organizer/dashboard`
- `ADMIN`：`/admin/reviews`

路由守卫规则：

- 未登录访问业务页面时跳转 `/login`。
- 已登录访问 `/login` 时跳转当前角色默认首页。
- 路由 `meta.roles` 与当前用户角色不匹配时显示 403 页面。
- Token 过期或后端返回 `40101` 时清空用户状态并回到登录页。

## 5. 状态管理

### 5.1 `userStore`

保存登录态和用户信息：

- `token`
- `userId`
- `name`
- `role`
- `studentNo`
- `phone`

动作：

- `login(payload)`
- `fetchMe()`
- `logout()`

### 5.2 `dictionaryStore`

缓存校区、场地、分类，用于活动筛选和创建表单：

- `campuses`
- `venues`
- `categories`
- `fetchCampuses()`
- `fetchVenues(campusId?)`
- `fetchCategories()`

### 5.3 `activityStore`

处理活动列表筛选条件和详情缓存：

- `filters`
- `activityList`
- `currentActivity`
- `fetchActivities()`
- `fetchActivityDetail(id)`

报名动作可以放在 `registrationApi` 中直接调用，避免 store 过度膨胀。

## 6. API 请求封装

`utils/request.ts` 统一封装 Axios：

- 请求拦截器自动注入 `Authorization: Bearer <token>`。
- 响应拦截器解包 `data`。
- `code !== 20000` 时统一弹出错误信息。
- 对 `40101` 执行登出和跳转。
- 对报名、审核等关键动作保留页面级错误提示，便于显示场地冲突、状态非法等业务原因。

## 7. 页面设计

### 7.1 登录页

- 用户名、密码输入框。
- 登录按钮显示 loading，避免重复提交。
- 登录后根据角色跳转。

### 7.2 学生端

#### 活动列表 `/student/activities`

- 顶部筛选：关键词、校区、分类、时间范围。
- 活动以卡片或紧凑列表展示，包含标题、时间、地点、分类、剩余名额、状态标签。
- 主要操作：查看详情。
- 列表页不堆叠复杂说明文字，筛选和结果优先。

#### 活动详情 `/student/activities/:id`

- 展示海报、标题、状态、时间、地点、报名截止、容量和详情。
- 根据 `registrationStatus` 展示不同按钮：
  - 未报名：`立即报名`
  - `ENROLLED`：`取消报名`、`查看签到码`
  - `WAITLISTED`：显示候补序号、`取消候补`
  - `CHECKED_IN`：显示签到时间
  - `CANCELLED`：允许重新报名，后端决定恢复为正选或候补
- 报名成功后即时刷新详情和列表余量。

#### 我的活动 `/student/my-registrations`

- 使用 Tabs 按状态分组：正选、候补、已签到、已取消。
- 每条记录展示活动标题、时间、地点、候补序号、签到时间。
- 正选和候补记录支持取消。

#### 签到码弹窗

- 点击后调用 `/registrations/{registrationId}/check-in-code`。
- 弹窗展示二维码和活动标题。
- 生成失败时提示当前状态不可签到。

### 7.3 组织者端

#### 工作台 `/organizer/dashboard`

- 显示我创建的活动数量、待审核数量、已发布数量、今日活动。
- 提供“创建活动”入口。

#### 活动管理 `/organizer/activities`

- 表格展示活动标题、时间、地点、状态、报名人数、审核原因。
- 操作：编辑草稿、提交审核、查看名单、取消活动。
- 状态标签颜色保持一致：草稿灰色、待审核蓝色、驳回红色、已发布绿色、已取消黑色。

#### 活动创建/编辑 `/organizer/activities/new`

- 表单字段：标题、分类、校区、场地、开始时间、结束时间、报名截止、容量、海报地址、详情。
- 选择校区后联动加载场地。
- 保存草稿与提交审核分成两个明确按钮。
- 提交前做前端校验：结束时间晚于开始时间、报名截止不晚于开始时间、容量大于 0。

#### 报名名单 `/organizer/activities/:id/registrations`

- 表格展示学生姓名、学号、电话、报名状态、候补序号、签到时间。
- 支持按状态筛选。
- 提供签到核销入口，可粘贴或扫描签到码。

### 7.4 管理员端

#### 审核中心 `/admin/reviews`

- 默认展示 `PENDING_REVIEW` 活动。
- 支持查看详情、通过、驳回。
- 驳回必须填写原因。

#### 资源管理

- 校区管理：列表、新增、编辑。
- 场地管理：按校区筛选，维护场馆名、房间号、容量。
- 分类管理：维护分类名称。
- 一期可以不做删除，避免外键引用导致误删问题；需要停用时二期增加状态字段。

#### 统计概览 `/admin/stats`

- 顶部数字卡：活动总数、待审核数、已发布数、报名总数、签到总数。
- 图表：校区活动数量、分类平均报名人数。
- 图表数据来自 `/stats/overview`, `/stats/campus-usage`, `/stats/category-popularity`。

## 8. 前端枚举

在 `utils/enums.ts` 中集中维护：

```ts
export const ActivityStatus = {
  DRAFT: 'DRAFT',
  PENDING_REVIEW: 'PENDING_REVIEW',
  REJECTED: 'REJECTED',
  PUBLISHED: 'PUBLISHED',
  ONGOING: 'ONGOING',
  FINISHED: 'FINISHED',
  CANCELLED: 'CANCELLED',
} as const

export const RegistrationStatus = {
  ENROLLED: 'ENROLLED',
  WAITLISTED: 'WAITLISTED',
  CANCELLED: 'CANCELLED',
  CHECKED_IN: 'CHECKED_IN',
  ABSENT: 'ABSENT',
} as const
```

标签文案、颜色和按钮状态都从枚举映射派生，避免各页面重复硬编码。

## 9. 交互细节

- 报名、取消、审核、签到等动作按钮必须有 loading 状态。
- 关键操作使用确认弹窗：取消报名、取消活动、审核驳回。
- 表单提交失败时保留用户已填写内容。
- 空列表提供简短空状态，例如“暂无待审核活动”。
- 所有表格保留分页。
- 移动端优先保证学生详情页和签到码弹窗可用；后台页面以 PC 和平板为主。

## 10. 前端验收清单

- 三种角色能登录并进入正确首页。
- 学生能筛选活动、查看详情、报名、候补、取消、查看签到码。
- 候补转正后学生在“我的活动”中看到状态更新为 `ENROLLED`。
- 组织者能创建草稿、提交审核、查看审核结果、查看名单并签到。
- 管理员能通过和驳回活动，维护校区、场地、分类。
- API 错误能以用户可理解的消息展示，不暴露后端异常堆栈。
