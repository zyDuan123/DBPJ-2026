# 校园活动报名系统需求说明书

## 1. 项目背景与目标
 在校园环境中，学生组织和学院部门频繁举办各类讲座、比赛及文娱活动。目前缺乏一个统一、高效的数字化平台来统筹管理活动场地、分类审核及报名流程。本项目旨在通过科学的数据库设计，实现一个支持多校区、多角色协作的校园活动管理系统。

## 2. 用户角色定义
 系统基于“单表继承”策略，将用户统一管理并划分为三个角色：
*  **学生 (Student)**：主要用户，负责浏览活动、按分类检索活动、提交报名申请。
*  **组织者 (Organizer)**：活动发起方，负责填写活动详情、选择校区场地、申请活动分类并发布活动。
*  **系统管理员 (Admin)**：平台维护者，负责录入校区与场地字典数据、审核组织者发布的活动申请。

## 3. 核心业务流程
1.   **资源初始化**：管理员提前录入校区（如邯郸、江湾等）及对应场地（如 H3106、光华楼）信息。
2.   **活动发布**：组织者创建活动，需指定活动标题、时间、分类、地点及人数上限。
3.   **活动审核**：管理员对申请中的活动进行审核，审核通过后活动对学生可见。
4.   **报名参与**：学生在人数上限范围内提交报名。系统需自动记录报名时间，并更新活动当前的报名人数。

##  4. 关键数据项与实体关系
*  **多校区场地管理**：校区与场地为 1:N 关系，每个场地必须归属于一个明确的校区。
* **活动归属逻辑**：
    *  一个活动必须关联一个确定的场地（1:N）。
    *  一个活动必须属于一个确定的分类（1:N）。
*  **双重用户关联**：活动实体同时关联组织者（发布者）和管理员（审核者），两个外键均引用用户表主键。
*  **报名机制**：学生与活动之间为 M:N 关系，通过报名记录表实现。一个学生对同一活动仅限报名一次。

##  5. 业务规则与数据库约束
为保证数据一致性，数据库层需实现以下约束：
*  **完整性约束**：所有表必须定义主键，外键关联必须开启级联或受限限制。
*  **非空约束 (NOT NULL)**：用户密码、活动标题、报名时间等核心字段不可为空。
*  **唯一性约束 (UNIQUE)**：学号、校区名、分类名称具有唯一性。
*  **检查约束 (CHECK)**：
    *  活动人数上限（`capacity_limit`）必须大于 0。
    *  当前已报名人数（`current_enrollment`）不得超过人数上限。
*  **默认值 (DEFAULT)**：报名记录的状态字段默认为 0（待审核）。

##  6. 查询场景支持
*  **基础查询**：学生查看个人所有报名记录。
* **多表连接**：查询指定校区下今日举办的所有活动及其详细场地房号。
* **聚合统计**：统计各分类活动的平均报名人数，辅助管理员决策.
