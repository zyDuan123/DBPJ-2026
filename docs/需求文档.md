# 校园活动报名系统需求说明书

## 1. 项目背景与目标

校园内讲座、比赛、志愿服务、文娱活动等活动频繁开展，但活动发布、场地管理、报名统计和现场签到往往分散在线下表格或临时群聊中，容易出现信息不一致、名额统计不准、场地冲突和审核流程不清晰等问题。

本项目旨在建设一个以数据库为核心的校园活动报名系统。一期目标是完成可开发、可联调、可演示的业务闭环：资源维护、活动发布、管理员审核、学生报名、满员候补、取消递补、现场签到和基础统计。

## 2. 一期范围

一期实现课程可落地版，不追求复杂运营能力。核心范围如下：

- 登录鉴权和三类角色权限控制。
- 校区、场地、分类基础数据维护。
- 组织者创建活动草稿并提交审核。
- 管理员审核活动，通过后学生可见，驳回时记录原因。
- 学生浏览、筛选、查看和报名活动。
- 名额已满时自动进入候补队列。
- 学生取消正选报名后，候补第一位自动转正。
- 学生生成签到码，组织者完成现场签到核销。
- 管理员查看基础统计。

二期扩展包括信用分、评价反馈、通知中心、审计日志、名单导出和文件上传。

## 3. 用户角色

### 3.1 学生 Student

学生是活动参与者，主要需求包括：

- 按关键词、校区、分类、时间筛选活动。
- 查看活动详情、剩余名额和报名截止时间。
- 报名活动，满员时进入候补。
- 取消自己的报名或候补。
- 查看个人活动记录和报名状态。
- 在现场出示签到码完成签到。

### 3.2 组织者 Organizer

组织者是活动发起方，主要需求包括：

- 创建和编辑活动草稿。
- 选择校区、场地、分类、活动时间和报名截止时间。
- 提交活动审核，查看审核状态和驳回原因。
- 查看自己活动的报名名单和候补名单。
- 核销学生签到码。
- 取消自己发布的活动。

### 3.3 管理员 Admin

管理员是平台维护者，主要需求包括：

- 维护校区、场地和活动分类。
- 审核组织者提交的活动。
- 处理场地时间冲突和不合规活动。
- 查看活动、报名、签到和资源使用统计。

## 4. 核心业务流程

### 4.1 资源初始化

管理员录入校区、场地和分类。每个场地必须归属一个校区，并设置容纳人数。同一校区内同一场馆和房间号不能重复。

### 4.2 活动发布与审核

1. 组织者创建活动草稿，状态为 `DRAFT`。
2. 组织者提交审核，状态变为 `PENDING_REVIEW`。
3. 系统校验活动时间、报名截止时间、人数上限和场地冲突。
4. 管理员审核通过后，活动状态变为 `PUBLISHED`，学生可见并可报名。
5. 管理员驳回后，活动状态变为 `REJECTED`，组织者可修改后再次提交。

### 4.3 报名与候补

1. 学生只能报名 `PUBLISHED` 且未超过报名截止时间的活动。
2. 活动未满员时，报名状态为 `ENROLLED`，占用名额。
3. 活动满员时，报名状态为 `WAITLISTED`，进入候补队列，不占用名额。
4. 学生对同一活动不能产生多条有效报名。
5. 学生取消正选报名后，系统自动将候补队首转为正选。

### 4.4 签到

1. 已正选报名的学生可以生成签到码。
2. 组织者或管理员核销签到码。
3. 签到成功后，报名状态变为 `CHECKED_IN`，并记录签到时间。
4. 重复签到应保持幂等，不重复写入异常数据。

## 5. 状态枚举

### 5.1 活动状态

| 状态 | 含义 |
| :--- | :--- |
| `DRAFT` | 草稿 |
| `PENDING_REVIEW` | 待审核 |
| `REJECTED` | 审核驳回 |
| `PUBLISHED` | 已发布 |
| `ONGOING` | 进行中 |
| `FINISHED` | 已结束 |
| `CANCELLED` | 已取消 |

### 5.2 报名状态

| 状态 | 含义 |
| :--- | :--- |
| `ENROLLED` | 正选报名成功 |
| `WAITLISTED` | 候补中 |
| `CANCELLED` | 已取消 |
| `CHECKED_IN` | 已签到 |
| `ABSENT` | 未签到，二期信用体系预留 |

## 6. 关键数据项

- 用户：角色、姓名、学号、手机号、密码。
- 校区：校区名、位置。
- 场地：场馆名、房间号、容量、所属校区。
- 分类：分类名称。
- 活动：标题、开始时间、结束时间、报名截止时间、人数上限、当前报名人数、状态、驳回原因、海报地址、详情、场地、分类、组织者、审核者。
- 报名记录：学生、活动、报名时间、报名状态、候补序号、签到时间。

## 7. 数据库约束要求

- 所有表必须定义主键。
- 外键必须保证引用完整性。
- 学号、手机号、校区名、分类名唯一。
- 同一校区内场馆名和房间号组合唯一。
- 同一学生对同一活动只能有一条报名记录。
- 活动结束时间必须晚于开始时间。
- 报名截止时间不得晚于活动开始时间。
- 活动人数上限必须大于 0。
- 当前报名人数不得超过人数上限。

## 8. 查询与统计场景

- 学生查询可报名活动列表。
- 学生查询个人报名记录。
- 组织者查询自己活动的报名名单。
- 管理员查询待审核活动。
- 管理员统计各校区活动数量和场地使用情况。
- 管理员统计各分类活动数量和平均报名人数。

## 9. 验收标准

- 三类角色登录后看到不同功能入口。
- 管理员能维护资源字典并审核活动。
- 组织者能创建、提交、查看和取消自己的活动。
- 学生能浏览、报名、候补、取消和查看签到码。
- 候补递补逻辑正确，取消正选后第一位候补自动转正。
- 签到核销能正确更新报名状态。
- API、前端、后端和数据库文档中的字段名与状态枚举保持一致。
