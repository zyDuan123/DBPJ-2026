# 校园活动报名系统：数据库设计与规范化分析报告

## 一、 关系模式定稿
基于 ER 图逻辑，系统共有 6 张核心关系表。主键以 `(PK)` 标识，外键以 `(FK)` 标识。

1.  **用户表 (User)**: `(用户ID (PK), 角色, 姓名, 学号 (Unique), 密码, 联系电话)`
    * *说明*：采用单表继承策略，通过“角色”字段区分学生、组织者与管理员。
2.  **校区表 (Campus)**: `(校区ID (PK), 校区名, 位置)`
3.  **场地表 (Venue)**: `(场地ID (PK), 场馆名, 房间号, 容纳人数, 校区ID (FK))`
4.  **分类表 (Category)**: `(分类ID (PK), 类别名称)` 
5.  **活动表 (Activity)**: `(活动ID (PK), 标题, 时间, 人数上限, 当前人数, 详情, 场地ID (FK), 分类ID (FK), 发布者ID (FK), 审核者ID (FK))` 
6.  **报名记录表 (Registration)**: `((学生ID (FK), 活动ID (FK)) (PK), 报名时间, 状态)`

## 二、 主键与外键约束说明 
1.  **完整性保障**：
    * 所有实体表均设置了自增或唯一的 ID 作为主键。
    * **1:N 关系**：通过在“多”端引入“一”端的主键作为外键实现逻辑关联。例如，`Venue` 表通过 `Campus_ID` 关联校区；`Activity` 表通过四个外键关联场地、分类及相关人员。
    * **M:N 关系**：`Registration` 作为中间表，其复合主键由 `Student_ID` 和 `Act_ID` 共同构成，确保单一学生对同一活动不可重复报名。

## 三、 业务规则与数据约束设计 
为了在数据库层面保障业务逻辑，我们设计了以下约束：
* **非空约束 (NOT NULL)**：所有主键、外键以及关键业务字段（如 `User` 的密码、`Activity` 的标题）均设为不可为空。
* **唯一约束 (UNIQUE)**：学号（`student_no`）与联系电话必须全局唯一。
* **检查约束 (CHECK)**：
    * `Venue.capacity` > 0：场内容纳人数必须为正数。
    * `Activity.capacity_limit` > 0：活动人数上限必须为正数。
    * **动态一致性**：`current_enrollment <= capacity_limit`（当前人数不得超过上限）。
* **默认值 (DEFAULT)**：`Registration.status` 默认为 0（待审核）。

## 四、 第三范式 (3NF) 规范化分析
1.  **满足 1NF**：所有属性均为原子项，如“位置”与“校区名”已严格分离。
2.  **满足 2NF**：消除了部分函数依赖。在 `Registration` 表中，报名时间和状态完全依赖于“学生ID+活动ID”的组合整体。
3.  **满足 3NF**：消除了传递函数依赖。
    * **规避更新异常**：活动表仅存 `Venue_ID`。若场地信息变更，仅需修改场地表，不会影响历史活动记录。
    * **规避插入异常**：校区与场地数据可独立于活动数据提前录入。

## 五、 反范式权衡与性能优化
* **设计决策**：我们在 `Activity` 表中冗余存储了 `current_enrollment`（当前报名人数）字段。
* **权衡理由**：虽然这不符合严格的范式要求（该值本应通过 `COUNT` 计算），但在“学生浏览活动墙”这一高频场景下，实时聚合计算会带来严重的性能瓶颈。
* **保障机制**：我们将通过后端事务（Transaction）控制，在 `Registration` 表发生增删时同步更新该冗余字段，以牺牲极小的存储空间换取查询性能的巨大提升。
