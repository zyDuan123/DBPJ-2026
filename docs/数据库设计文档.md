# 校园活动报名系统：数据库设计与规范化分析报告

## 一、设计目标

数据库设计服务于“课程可落地版 + 二期增强”功能闭环：用户登录、活动发布审核、活动报名、满员候补、取消递补、现场签到、活动评价反馈、信用分、基础统计和资源字典维护。当前保留 8 张核心表，通知、审计等能力继续作为后续扩展。

目标数据库版本为 MySQL 8.4.8 LTS。

## 二、核心关系模式

### 1. 用户表 `User`

`(user_id PK, role, username, student_no UNIQUE, password, phone UNIQUE)`

说明：

- 采用单表继承策略，通过 `role` 区分学生、组织者、管理员。
- `student_no` 主要用于学生账号，可为空但非空时必须唯一。
- `password` 存储加密后的密码，不保存明文。

### 2. 校区表 `Campus`

`(campus_id PK, campus_name UNIQUE, location)`

说明：校区是场地的上级字典。

### 3. 场地表 `Venue`

`(venue_id PK, venue_name, room_number, capacity, campus_id FK)`

说明：

- 每个场地必须属于一个校区。
- 建议增加唯一约束 `UNIQUE(campus_id, venue_name, room_number)`，避免同一校区重复录入同一房间。

### 4. 分类表 `Category`

`(category_id PK, category_name UNIQUE)`

说明：活动分类字典，如学术讲座、比赛竞赛、文娱活动、志愿服务。

### 5. 活动表 `Activity`

`(activity_id PK, title, start_time, end_time, enroll_deadline, capacity_limit, current_enrollment, status, reject_reason, poster_url, description, venue_id FK, category_id FK, organizer_id FK, admin_id FK)`

说明：

- `start_time` 与 `end_time` 描述活动举办时间。
- `enroll_deadline` 描述报名截止时间。
- `current_enrollment` 是冗余字段，只统计占用名额的报名记录。
- `status` 使用活动状态枚举。
- `reject_reason` 用于管理员驳回活动。
- `poster_url` 用于前端活动列表和详情展示。
- `organizer_id` 与 `admin_id` 均引用 `User(user_id)`。

### 6. 报名记录表 `Registration`

`(registration_id PK, student_id FK, activity_id FK, registration_time, status, queue_no, check_in_time)`

说明：

- 使用独立主键 `registration_id` 便于签到码、列表操作和接口引用。
- 增加唯一约束 `UNIQUE(student_id, activity_id)`，从数据库层防止重复报名。
- `status` 使用报名状态枚举。
- `queue_no` 仅候补记录使用。
- `check_in_time` 仅签到后写入。

### 7. 活动评价表 `ActivityFeedback`

`(feedback_id PK, registration_id FK UNIQUE, activity_id FK, student_id FK, rating, content, created_at, updated_at)`

说明：

- 评价与报名记录绑定，一条报名记录最多提交一条评价。
- 只有已签到记录允许评价，业务层负责校验 `Registration.status = CHECKED_IN`。
- `rating` 使用 1-5 分，`content` 保存文字反馈。
- 冗余保存 `activity_id` 和 `student_id`，便于组织者反馈看板、管理员统计和学生查询。

### 8. 信用流水表 `CreditRecord`

`(record_id PK, student_id FK, activity_id FK, registration_id FK, change_value, reason_type, reason, operator_id FK, created_at)`

说明：

- 学生信用分按 `100 + SUM(change_value)` 实时计算，不在 `User` 表冗余保存。
- `CHECK_IN` 表示完成签到，当前规则为 `+1`。
- `ABSENT` 表示活动结束后未签到，当前规则为 `-10`。
- `MANUAL_ADJUST` 预留给后续管理员人工调整。
- `UNIQUE(reason_type, registration_id)` 避免同一报名记录重复写入同类信用流水。

## 三、状态枚举

### 1. 活动状态 `Activity.status`

| 状态 | 含义 |
| :--- | :--- |
| `DRAFT` | 草稿 |
| `PENDING_REVIEW` | 待审核 |
| `REJECTED` | 审核驳回 |
| `PUBLISHED` | 已发布，可报名 |
| `ONGOING` | 进行中 |
| `FINISHED` | 已结束 |
| `CANCELLED` | 已取消 |

### 2. 报名状态 `Registration.status`

| 状态 | 含义 | 是否占用名额 |
| :--- | :--- | :---: |
| `ENROLLED` | 正选报名成功 | 是 |
| `WAITLISTED` | 候补中 | 否 |
| `CANCELLED` | 已取消 | 否 |
| `CHECKED_IN` | 已签到 | 是 |
| `ABSENT` | 未签到，二期信用体系预留 | 否 |

一期实现 `ENROLLED`, `WAITLISTED`, `CANCELLED`, `CHECKED_IN`；二期已实现 `ABSENT`，用于信用分与缺勤处理。

## 四、主键、外键与完整性约束

### 1. 主键

- `User.user_id`
- `Campus.campus_id`
- `Venue.venue_id`
- `Category.category_id`
- `Activity.activity_id`
- `Registration.registration_id`
- `ActivityFeedback.feedback_id`
- `CreditRecord.record_id`

### 2. 外键

- `Venue.campus_id -> Campus.campus_id`
- `Activity.venue_id -> Venue.venue_id`
- `Activity.category_id -> Category.category_id`
- `Activity.organizer_id -> User.user_id`
- `Activity.admin_id -> User.user_id`
- `Registration.student_id -> User.user_id`
- `Registration.activity_id -> Activity.activity_id`
- `ActivityFeedback.registration_id -> Registration.registration_id`
- `ActivityFeedback.activity_id -> Activity.activity_id`
- `ActivityFeedback.student_id -> User.user_id`
- `CreditRecord.student_id -> User.user_id`
- `CreditRecord.activity_id -> Activity.activity_id`
- `CreditRecord.registration_id -> Registration.registration_id`
- `CreditRecord.operator_id -> User.user_id`

### 3. 唯一约束

- `User.student_no`
- `User.phone`
- `Campus.campus_name`
- `Category.category_name`
- `Venue(campus_id, venue_name, room_number)`
- `Registration(student_id, activity_id)`
- `ActivityFeedback(registration_id)`
- `CreditRecord(reason_type, registration_id)`

### 4. 检查约束

- `User.role IN ('STUDENT', 'ORGANIZER', 'ADMIN')`，若使用数字编码则需在文档和代码中统一映射。
- `Venue.capacity > 0`。
- `Activity.capacity_limit > 0`。
- `Activity.current_enrollment >= 0`。
- `Activity.current_enrollment <= Activity.capacity_limit`。
- `Activity.end_time > Activity.start_time`。
- `Activity.enroll_deadline <= Activity.start_time`。
- `Activity.status` 仅允许活动状态枚举。
- `Registration.status` 仅允许报名状态枚举。
- `ActivityFeedback.rating BETWEEN 1 AND 5`。
- `CreditRecord.reason_type IN ('CHECK_IN', 'ABSENT', 'MANUAL_ADJUST')`。

## 五、关键业务一致性

### 1. 报名人数冗余字段

`Activity.current_enrollment` 是为了优化活动列表查询而保留的反范式字段。它只统计占用名额的报名记录：

- `ENROLLED`
- `CHECKED_IN`

不统计：

- `WAITLISTED`
- `CANCELLED`
- `ABSENT`

所有会影响该字段的操作必须在后端事务内完成。

### 2. 报名与候补

报名事务逻辑：

1. 锁定目标活动行。
2. 校验活动状态、报名截止时间和重复报名。
3. 若 `current_enrollment < capacity_limit`，写入 `ENROLLED` 并增加当前人数。
4. 否则写入 `WAITLISTED`，分配 `queue_no`，不增加当前人数。

取消正选报名时：

1. 原报名改为 `CANCELLED`。
2. 当前人数减 1。
3. 查询最早候补记录并转为 `ENROLLED`。
4. 若候补转正成功，当前人数加 1。

### 3. 场地时间冲突

同一场地在相互重叠的时间段内，不允许存在多个处于 `PENDING_REVIEW`、`PUBLISHED` 或 `ONGOING` 的活动。

时间重叠判断：

```sql
new_start < existing_end AND new_end > existing_start
```

建议后端 Service 先做冲突查询并返回友好错误，数据库触发器或存储过程作为兜底。

### 4. 签到

签到只允许 `ENROLLED` 报名记录变更为 `CHECKED_IN`，并写入 `check_in_time`。重复签到应保持幂等。

### 5. 评价反馈

评价只允许已签到学生提交。学生重复提交同一活动评价时，不新增记录，而是更新原评价的评分、内容和 `updated_at`。

组织者只能查看自己活动的反馈明细；管理员可以查看所有活动反馈和全站评价概览。

### 6. 信用分与缺勤

活动结束后，组织者或管理员可以将仍为 `ENROLLED` 的报名记录批量标记为 `ABSENT`。该操作必须在事务中完成：

1. 锁定目标活动。
2. 校验活动已结束且操作者有权限。
3. 将未签到正选记录从 `ENROLLED` 更新为 `ABSENT`。
4. 减少活动 `current_enrollment`。
5. 写入 `CreditRecord`，缺勤扣 `-10`。

学生完成签到时写入 `CHECK_IN` 信用流水，当前规则为 `+1`。信用流水通过唯一约束保持幂等。

## 六、规范化分析

### 1. 第一范式

所有字段均为原子属性。例如活动时间拆分为 `start_time` 和 `end_time`，场地通过 `venue_name` 与 `room_number` 描述，不在活动表中重复保存校区名称。

### 2. 第二范式

所有非主属性完全依赖各自表的主键。报名表改用独立主键后，`student_id + activity_id` 通过唯一约束表达业务唯一性，报名时间、状态、候补序号均依赖报名记录本身。

### 3. 第三范式

活动表只保存 `venue_id` 和 `category_id`，不冗余校区名、场地名、分类名，避免字典信息变更造成多处更新异常。

### 4. 反范式说明

`current_enrollment` 是明确的性能型冗余字段。它可以减少学生首页活动墙频繁执行 `COUNT` 聚合的成本，但必须由后端事务和数据库约束共同保证一致性。

## 七、索引建议

- `Activity(status, start_time)`：优化学生首页和活动列表。
- `Activity(venue_id, start_time, end_time)`：优化场地冲突检测。
- `Activity(organizer_id, status)`：优化组织者活动管理。
- `Registration(activity_id, status, queue_no)`：优化名单查询和候补递补。
- `Registration(student_id, status)`：优化学生“我的活动”。
- `ActivityFeedback(activity_id, rating)`：优化单活动反馈统计。
- `ActivityFeedback(student_id, activity_id)`：优化学生个人评价查询。
- `CreditRecord(student_id, created_at)`：优化学生信用流水。
- `CreditRecord(activity_id, reason_type)`：优化活动缺勤与信用统计。
- `Venue(campus_id)`：优化校区下场地查询。

## 八、一期与二期边界

一期保留 6 张核心表并完成报名签到闭环；二期已增加 `ActivityFeedback` 和 `CreditRecord`，用于活动评价、反馈看板、信用分与缺勤处理。后续可按需增加：

- `Notification`：站内消息。
- `Audit_Log`：敏感操作审计。

这些扩展不影响现有核心表结构，只通过外键关联 `User`、`Activity` 或 `Registration`。
