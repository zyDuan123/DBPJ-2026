# 校园活动报名系统后端开发文档

## 1. 后端目标

后端一期负责提供稳定的 REST API、角色权限控制、活动审核状态流转、报名并发一致性、候补递补、签到核销和基础统计。所有核心业务规则以后端和数据库约束为准，前端校验只作为体验优化。

## 2. 技术栈

- Java 17 或课程环境指定版本
- Spring Boot
- Spring Web
- 自定义 HMAC Token 鉴权
- MyBatis-Plus
- MySQL 8.4.8 LTS
- Lombok
- Bean Validation
- Druid 或 HikariCP
- JUnit 5

- DTO 接收请求参数，VO 返回前端展示数据。

## 4. 统一响应与异常

统一响应：

```java
public class Result<T> {
    private Integer code;
    private String message;
    private T data;
}
```

核心业务码：

| 业务码 | 名称 | 使用场景 |
| :--- | :--- | :--- |
| `20000` | `SUCCESS` | 成功 |
| `40001` | `PARAM_ERROR` | Bean Validation 或参数解析失败 |
| `40101` | `UNAUTHORIZED` | 未登录、Token 过期 |
| `40301` | `FORBIDDEN` | 权限不足 |
| `40401` | `NOT_FOUND` | 数据不存在 |
| `40901` | `DUPLICATE_REGISTRATION` | 重复报名冲突 |
| `40902` | `VENUE_CONFLICT` | 场地时间冲突 |
| `40903` | `INVALID_STATUS` | 当前状态不允许操作 |
| `40904` | `ENROLL_CLOSED` | 活动不可报名 |
| `50000` | `SERVER_ERROR` | 未预期异常 |

`GlobalExceptionHandler` 负责将业务异常、校验异常、数据库约束异常统一转换为上述结构。

## 5. 权限控制

用户角色：

- `STUDENT`
- `ORGANIZER`
- `ADMIN`

当前实现：

- 登录接口按账号查询用户，使用 PBKDF2 哈希校验密码，数据库不保存明文密码。
- Token 使用 HMAC-SHA256 签名，载荷包含 `userId` 和过期时间。
- `AuthInterceptor` 解析 Token 并写入 `AuthContext`。
- Service 层通过 `Access.require(...)` 控制角色。
- 涉及资源归属的接口必须二次校验，例如组织者只能修改自己的活动。

权限要点：

- 学生只能报名、取消自己的报名、生成自己的签到码。
- 组织者只能管理自己创建的活动和名单。
- 管理员可以审核活动、维护字典、查看统计。

## 7. 活动模块实现

### 7.1 创建草稿

创建活动时写入：

- `status = DRAFT`
- `current_enrollment = 0`
- `organizer_id = 当前用户 ID`

校验：

- 标题非空。
- `endTime > startTime`。
- `enrollDeadline <= startTime`。
- `capacityLimit > 0`。
- 场地、分类存在。

### 7.2 提交审核

只有活动创建者可以提交自己的 `DRAFT` 或 `REJECTED` 活动。提交前再次执行基础校验和场地冲突校验，成功后状态变为 `PENDING_REVIEW`，清空旧的 `reject_reason`。

### 7.3 管理员审核

只有 `PENDING_REVIEW` 可审核。

- `APPROVED`：校验场地冲突后状态变为 `PUBLISHED`，写入 `admin_id`。
- `REJECTED`：状态变为 `REJECTED`，写入 `reject_reason`，驳回原因必填。

### 7.4 场地冲突

冲突条件：

- 同一 `venue_id`。
- 目标活动状态属于 `PENDING_REVIEW`, `PUBLISHED`, `ONGOING`。
- 时间区间重叠：`newStart < oldEnd && newEnd > oldStart`。

Service 层必须先查询并返回友好错误；数据库触发器或唯一化辅助约束作为兜底。

## 8. 报名模块实现

报名、取消和候补递补必须使用 `@Transactional`。

### 8.1 报名

流程：

1. 查询活动并加行锁，避免并发超卖。
2. 校验活动状态为 `PUBLISHED` 或按当前时间可报名。
3. 校验未超过 `enroll_deadline`。
4. 查询当前学生对该活动是否已有报名记录。
5. 已有 `ENROLLED` 或 `WAITLISTED` 时直接返回该记录。
6. 已有 `CANCELLED` 时可复用记录并更新状态。
7. 若正选人数未满，状态设为 `ENROLLED`，`current_enrollment + 1`。
8. 若已满，状态设为 `WAITLISTED`，`queue_no = 当前最大候补序号 + 1`。

数据库层使用 `UNIQUE(student_id, activity_id)` 防重复。

### 8.2 取消报名

流程：

1. 查询报名记录并校验操作者身份。
2. 若状态为 `WAITLISTED`，直接改为 `CANCELLED`。
3. 若状态为 `ENROLLED`，改为 `CANCELLED`，`current_enrollment - 1`。
4. 查找同活动最早的 `WAITLISTED` 记录，将其转为 `ENROLLED`，清空或保留 `queue_no` 均可，但接口要返回最新状态。
5. 若成功递补，`current_enrollment + 1`，最终人数保持不变。

### 8.3 签到

签到码可用 JWT 或后端签名字符串实现，至少包含 `registrationId` 和过期时间。

核销流程：

1. 解析并校验签到码。
2. 查询报名记录和活动。
3. 校验当前组织者是活动创建者，或当前用户是管理员。
4. 只有 `ENROLLED` 可签到；已 `CHECKED_IN` 时幂等返回。
5. 更新状态为 `CHECKED_IN`，写入 `check_in_time`。

## 9. 字典与统计模块

### 9.1 字典

字典包括校区、场地、分类。

- 校区名唯一。
- 同一校区内 `venue_name + room_number` 唯一。
- 分类名唯一。
- 一期优先实现新增、查询、编辑；删除可以暂缓，避免破坏已有关联数据。

### 9.2 统计

统计接口只做基础聚合：

- `/stats/overview`：活动数、待审核数、已发布数、报名数、签到数。
- `/stats/campus-usage`：按校区统计活动数和场地数。
- `/stats/category-popularity`：按分类统计活动数和平均报名人数。

统计查询不参与核心事务，不应影响报名接口性能。

## 11. 数据库事务与一致性

- `current_enrollment` 是冗余字段，只统计 `ENROLLED` 和 `CHECKED_IN` 这类占用名额的记录。
- 所有修改报名状态并影响人数的操作必须在同一事务内完成。
- 并发报名推荐对活动行执行 `SELECT ... FOR UPDATE` 或 MyBatis-Plus 等价写法。
- 数据库检查约束保证 `current_enrollment <= capacity_limit`。
- Service 层不能只依赖前端传入的状态，必须重新读取数据库状态。

## 12. 测试要求

### 12.1 单元测试

- 活动时间校验。
- 活动状态流转校验。
- 报名状态流转校验。
- 权限判断工具方法。

### 12.2 集成测试

- 学生报名成功，人数加 1。
- 满员后报名进入候补。
- 重复报名返回原记录，不新增数据。
- 正选取消后候补第一位自动转正。
- 组织者不能审核活动。
- 学生不能访问管理员统计接口。
- 场地时间冲突返回 `VENUE_CONFLICT`。
- 已签到记录重复签到保持幂等。

## 13. 二期预留

- 审计日志：记录管理员与组织者的重要操作。
- Excel 导出：组织者导出报名名单。
